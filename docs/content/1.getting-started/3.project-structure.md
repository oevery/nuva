下载模板后，你最常接触的是 `app`、`server` 和 `shared`。只要先知道这三类内容分别放哪里，就足够开始业务开发。

```txt
app/                 # 页面、组件、前端 API 调用
server/              # 服务端接口
shared/              # 前后端共享类型和 schema
nuxt.config.ts       # 项目配置
```

## `app/pages`

页面路由和业务页面入口都放在这里。用户能看到的业务流程，通常从这里开始落地。

## `app/components`

放复用组件。优先服务当前项目的业务场景。

## `app/composables/apis`

放业务 API 封装和组合式数据获取逻辑。推荐同一业务域收敛到一个文件，页面只负责调用。

## `app/utils/http/hooks.ts`

这里是你调整请求行为的主要入口，比如追加请求头和统一错误处理。登录态失效和 401 跳转建议参考 [认证与权限](/auth)。

## `server/api`

放服务端接口实现。接口协议尽量和 `shared` 里的类型、schema 保持一致。

## `shared`

放前后端共享类型、valibot schema 和通用协议定义。客户端和服务端都从这里读取同一份定义。

## `nuxt.config.ts`

这里配置项目自己的模块和运行时参数，通常只在你要调整请求、认证或构建行为时才会改它。

## 最常见的判断方式

- 这是页面展示逻辑吗？放 `app`。
- 这是服务端接口或服务端辅助逻辑吗？放 `server`。
- 这是前后端都要用到的类型或 schema 吗？放 `shared`。

## 推荐的放置原则

- 页面和页面级状态放 `app/pages`。
- 复用组件放 `app/components`。
- 业务 API 放 `app/composables/apis`。
- 请求行为定制放 `app/utils/http/hooks.ts`。
- 服务端接口放 `server/api`。
- 共享类型和 schema 放 `shared`。

如果你不确定代码该放哪里，优先放到项目自己的 `app`、`server` 和 `shared` 里；大多数业务开发不会先碰到底层实现。

如果你想看这些部分之间是怎么协作的，再去看 [整体架构](/getting-started/architecture)。
