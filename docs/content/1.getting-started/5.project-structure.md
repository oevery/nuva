title: 项目结构
description: 需要写页面、接口、共享类型和配置时，先知道该从哪里开始。
---

下载模板后，你最常接触的是 `app`、`server`、`shared`、`nuva.config.ts` 和 `nuxt.config.ts`。

```txt
app/                 # 页面、组件、前端 API 调用
server/              # 服务端接口和服务端工具
shared/              # 前后端共享类型和 schema
nuva.config.ts       # Nuva 领域配置
nuxt.config.ts       # Nuxt 项目配置
```

## `app`

`app` 放前端应用代码。常见位置：

- `app/pages`：页面路由和业务页面入口。
- `app/components`：业务组件和复用组件。
- `app/composables/apis`：业务 API 封装和组合式数据获取逻辑。
- `app/utils/http/hooks.ts`：请求头、错误提示、401/403 等请求行为扩展。

## `server`

`server` 放 Nuxt server API 和服务端辅助逻辑。接口协议尽量和 `shared` 中的类型、schema 保持一致。

## `shared`

`shared` 放前后端都要使用的类型、valibot schema 和通用协议定义。客户端和服务端都通过 `#shared` 引用同一份定义。

## `nuva.config.ts`

`nuva.config.ts` 维护 Nuva 的请求、认证、权限和远程 resolver 配置。它是业务领域配置，不应该被拆成多套平行入口。

## `nuxt.config.ts`

`nuxt.config.ts` 维护 Nuxt 自身行为，例如 `extends`、模块、runtime config 和构建配置。

## 最常见判断方式

- 这是页面展示或交互逻辑吗？放 `app`。
- 这是服务端接口或服务端辅助逻辑吗？放 `server`。
- 这是前后端都需要的类型或 schema 吗？放 `shared`。
- 这是 Nuva 的请求、认证、权限配置吗？放 `nuva.config.ts`。
- 这是 Nuxt 模块、部署或构建行为吗？放 `nuxt.config.ts`。
