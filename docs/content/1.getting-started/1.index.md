Nuva 适合已经知道自己要做业务页面、接口和表单，但不想从零搭一套请求、认证和校验约定的项目。

对大多数使用者来说，第一次接入 Nuva 不需要先理解内部目录角色。更重要的是先完成第一次启动，再确认默认请求、认证和共享 schema 是否符合你的项目。

## 这套文档怎么读最省时间

- 如果你刚接项目：先读 [快速开始](/getting-started/quick-start)。
- 如果你想改接口协议：直接读 [请求体系](/request)。
- 如果你需要登录与页面保护：直接读 [认证与权限](/auth)。
- 如果你要新增表单或接口：直接读 [业务实现](/implementation)。

## 第一次接入时，推荐按这个顺序做

1. 先下载模板并启动项目。
2. 确认默认页面、接口和请求配置可以正常工作。
3. 根据你的项目，决定先接认证还是先写业务接口。
4. 开始补共享 schema、服务端接口和页面表单。

这个顺序的目标很明确：先确认默认能力能跑，再逐项替换成你自己的业务实现，而不是一开始就重构模板。

## 适合什么项目

- 需要 Nuxt SSR 或全栈能力的中后台、运营工具和 BFF 项目。
- 接口较多，需要统一请求、错误处理和登录态管理的项目。
- 希望前后端共用 schema、类型和接口协议的项目。
- 想先拿到一个能跑起来的起点，再逐步替换成自己业务实现的项目。

## 你第一次打开项目时，先看什么

1. 用 `giget` 下载 `template`。
2. 启动项目，先确认默认流程能跑通。
3. 看请求默认配置是否匹配你的接口协议。
4. 再决定要先接认证，还是先开始写业务接口和表单。

## 推荐阅读顺序

1. [快速开始](/getting-started/quick-start)：使用 `giget` 下载模板并启动项目。
2. [请求体系](/request)：配置 alova、响应协议、错误处理和业务 API。
3. [认证与权限](/auth)：配置 Auth、路由保护、401 处理和 Better Auth。
4. [共享类型与 schema](/capabilities/shared-schema)：了解前后端共用规则的方式。
5. [整体架构](/getting-started/architecture)：了解页面、接口、共享 schema 和请求层之间的关系。
6. [业务实现](/implementation)：新增接口、API composable、页面和表单。
