---
title: Agent 工作流
description: 让 AI 新增业务功能时遵循 Nuva 的文件顺序和验证方式。
---

让 AI 在 Nuva 项目里工作时，推荐明确告诉它遵循以下顺序。

## 新增业务功能

```txt
1. 先读取相关 shared 类型和 schema。
2. 新增或修改 server/api。
3. 新增或修改 app/composables/apis。
4. 最后修改 app/pages 或 app/components。
5. 运行 typecheck、lint 或对应构建命令。
```

## 新增表单

```txt
1. 在 shared/api 中定义 valibot schema。
2. 服务端接口使用同一份 schema 校验请求体。
3. 前端表单使用 @vee-validate/valibot。
4. 服务端字段错误回填到表单。
```

## 新增权限

```txt
1. 定义权限点，优先使用 resource:action。
2. 页面 meta 保护路由。
3. NuvaCan 或 usePermission 控制局部 UI。
4. server utils 保护真实 API。
```

## 验证建议

- 修改业务页面、接口或 shared schema 后，运行业务项目自己的 `pnpm typecheck` 和 `pnpm build`。
- 修改请求、认证或权限配置后，同时验证登录、401/403、受保护页面和服务端 API。
- 修改表单 schema 后，同时验证前端字段错误和服务端字段错误回填。
- 如果你是在维护 Nuva 仓库本身，验证策略见 [测试与检查](/maintenance/testing)。

## 不建议让 AI 做什么

- 未确认业务协议时重构目录结构。
- 在页面里重复实现请求协议。
- 只隐藏按钮但不保护服务端 API。
- 在没有明确需求时引入第二套请求或认证方案。
