这里的升级策略面向 Nuva 维护者。用户项目升级 `@oevery/nuva` 时，只需要查看 release notes 和对应 Reference。

## Core 升级策略

修改 core 时需要判断：

- 是否影响已有导入路径。
- 是否影响 `nuva.config.ts` 字段。
- 是否影响 `runtimeConfig.public.nuva` 结构。
- 是否影响 template 的默认实现。
- 是否影响文档中的 Recipes。

## Template 同步策略

模板是起点，不是业务项目的自动覆盖源。维护 template 时优先同步：

- `nuva.config.ts` 推荐配置。
- `app/utils/http/hooks.ts` 默认 hooks 用法。
- `shared` 中通用协议类型。
- 认证和权限示例页面。
- getting started 中的 first feature 流程。

## 破坏性变更处理

如果出现配置字段或导出路径变化：

1. 更新 [导出入口](/reference/exports)。
2. 更新 [Nuva Config](/reference/nuva-config)。
3. 更新相关 Guide、Recipe 和 template 示例。
4. 在 release notes 中明确迁移方式。
5. 跑 typecheck、test 和 build。

## 保持用户文档稳定

用户文档应尽量描述稳定路径。如果某个能力仍在探索中，优先放在开发维护或内部说明，不要放到用户必读路径。
