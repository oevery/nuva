---

Nuva 模板使用 alova 作为业务请求入口。大多数项目只需要先确认 3 件事：API 基础地址、响应是否自动解包、成功码有哪些。

## 这页帮你解决什么

- 在哪里配置请求基础能力。
- `useHttpClient()` 为什么能统一处理响应协议。
- 业务 API 该怎么组织，才不会把页面和请求逻辑耦合在一起。

## 配置入口

请求配置位于 `runtimeConfig.public.nuva.api`。认证与权限配置见 [认证与权限](/auth)。

```ts
import type { NuvaApiConfig } from '@oevery/nuva/config'

const api = {
  baseURL: '/api',
  envelopeUnwrap: true,
  successCodes: '0,200,SUCCESS',
} satisfies NuvaApiConfig

export default defineNuxtConfig({
  runtimeConfig: {
    public: {
      nuva: {
        api,
      },
    },
  },
})
```

## 响应协议

常见中后台接口会返回统一包裹结构：

```ts
export interface ApiResponse<T = unknown> {
  code: number
  message: string
  data: T
}
```

当 `envelopeUnwrap` 为 `true` 时，`useHttpClient()` 会校验 `code`，成功后返回 `data`。

这意味着页面和业务 composable 通常不需要反复写 `response.data.data` 这一层解包逻辑。

## 单接口配置

单个接口可以通过 alova `meta` 覆盖全局行为。

```ts
http.Get('/profile', {
  meta: {
    ignoreToken: true,
    responseType: 'json',
    envelopeUnwrap: true,
    successCodes: '0,200',
  },
})
```

常用字段：

- `ignoreToken`：跳过 token 注入。
- `responseType`：指定响应解析类型，支持 `json`、`text`、`blob`、`arrayBuffer`、`raw`。
- `envelopeUnwrap`：单接口是否解包 `{ code, message, data }`。
- `successCodes`：单接口成功码。

## 业务 API 放哪里

业务 API 推荐放在：

```txt
app/composables/apis/*.ts
```

同一业务域尽量收敛到一个文件：

- `useXxxApi` 放接口方法。
- `useXxxTable` 放分页、筛选等状态组合。
- 页面只负责调用，不直接拼请求细节。

## 错误处理

请求错误可以在 `app/utils/http/hooks.ts` 中统一处理。普通接口错误放在请求体系里处理；登录态失效和 401 跳转建议交给 [认证与权限](/auth/route-protection)。

如果你在项目里已经有现成的后端协议，优先改 `runtimeConfig.public.nuva.api` 和 `app/utils/http/hooks.ts`，不要在每个页面单独兼容。
