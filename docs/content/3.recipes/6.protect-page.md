---
title: 保护页面
description: 使用 Auth core 的 route middleware 保护页面，并声明角色、权限和作用域要求。
---

## 目标

让需要登录或权限的页面统一经过 Nuva route middleware，而不是在每个页面手写跳转判断。

## 前置条件

启用 Auth core：

```ts
export default defineNuxtConfig({
  modules: ['@oevery/nuva/auth'],
})
```

配置登录页和保护策略：

```ts
import { defineNuvaConfig } from '@oevery/nuva/config'

export default defineNuvaConfig({
  auth: {
    provider: 'token',
    loginPath: '/login',
    homePath: '/',
    global: true,
    publicRoutes: ['/login'],
  },
})
```

## 默认受保护

`global: true` 时，页面默认受保护。公开页面显式声明：

```ts
definePageMeta({
  auth: false,
})
```

## 单页受保护

`global: false` 时，页面默认公开。需要保护的页面声明：

```ts
definePageMeta({
  auth: true,
})
```

## 页面权限

```ts
definePageMeta({
  auth: {
    roles: ['admin'],
    permissions: ['profile:read'],
    scopes: ['departmentId'],
    roleMode: 'any',
    permissionMode: 'all',
  },
})
```

无权限时会跳转到 `auth.permission.forbiddenPath`，默认是 `/403`。

## 登录后回跳

未登录访问受保护页面时，会跳转：

```txt
/login?redirect=/profile
```

登录成功后调用：

```ts
await useAuth().afterLogin()
```

## 常见问题

- 公开页面不要同时写 `auth: false` 和页面级权限要求。
- 页面守卫只负责能不能进入页面，按钮和接口仍要单独做权限控制。
- 服务端 API 必须使用 server utils 做真实拦截。
