---
title: Auth adapter 开发
description: 为 Auth.js、Supabase、企业 SSO 或自研 session 接入 Nuva Auth core。
---

Nuva Auth core 不直接绑定具体认证 SDK。除内置 token adapter 和 `@oevery/nuva/better-auth` 外，第三方认证方案可以通过 `@oevery/nuva/auth/adapter` 注册 adapter，然后继续复用 `useAuth()`、`usePermission()` 和 route middleware。

## 最小 adapter

adapter 是一个运行时 factory。factory 内可以调用 Nuxt composable，因为它会在运行时由业务代码或插件触发。

```ts
// app/plugins/custom-auth-adapter.ts
import { defineAuthAdapter, registerAuthAdapter } from '@oevery/nuva/auth/adapter'

export default defineNuxtPlugin(() => {
  registerAuthAdapter('custom-session', defineAuthAdapter(() => {
    const user = useState('custom-auth:user', () => null)
    const ready = useState('custom-auth:ready', () => false)

    return {
      user: computed(() => user.value),
      ready: computed(() => ready.value),
      isAuthenticated: computed(() => !!user.value),
      async ensureAuthenticated() {
        if (!ready.value) {
          user.value = await $fetch('/api/session')
          ready.value = true
        }

        return !!user.value
      },
      async logout() {
        await $fetch('/api/logout', { method: 'POST' })
        user.value = null
        ready.value = true
      },
    }
  }))
})
```

对应配置：

```ts
export default defineNuvaConfig({
  auth: {
    enabled: true,
    provider: 'custom-session',
  },
})
```

如果 `provider` 不是 `token`，项目必须在运行时注册同名 adapter。否则 `useAuth()`、route middleware 或 `usePermission()` 会抛出未注册错误。

adapter 的 `name` 由 registry 根据 `registerAuthAdapter(name, factory)` 注入；factory 返回值里不要重复写 `name`。

## 权限 adapter

如果认证 provider 也负责权限，设置 `permission.provider: 'adapter'`。这样 `usePermission()` 会把权限读取和 `canAsync()` 交给当前 auth adapter。

```ts
registerAuthAdapter('custom-session', defineAuthAdapter(() => {
  const roles = useState<string[]>('custom-auth:roles', () => [])

  return {
    user: computed(() => null),
    ready: computed(() => true),
    isAuthenticated: computed(() => true),
    logout: () => {},
    permission: {
      loaded: computed(() => true),
      state: computed(() => ({
        roles: roles.value,
        permissions: [],
        scope: {},
        dataAccess: { type: 'self' },
        source: 'adapter',
      })),
      canState(permission) {
        return permission === 'dashboard:view' ? 'allow' : 'unknown'
      },
      async canAsync(permission, context) {
        return await $fetch('/api/permission/check', {
          method: 'POST',
          body: { permission, context },
        })
      },
    },
  }
}))
```

```ts
export default defineNuvaConfig({
  auth: {
    enabled: true,
    provider: 'custom-session',
    permission: {
      provider: 'adapter',
    },
  },
})
```

## 模块封装

正式 provider 建议封装成 Nuxt module，而不是要求业务项目手写插件。模块中先安装 Auth core，再注册自己的运行时插件。

```ts
import { addPlugin, createResolver, defineNuxtModule, installModule } from '@nuxt/kit'

export default defineNuxtModule({
  meta: {
    name: '@acme/nuva-custom-auth',
    configKey: 'nuvaAuth',
  },
  async setup(options) {
    const resolver = createResolver(import.meta.url)

    await installModule('@oevery/nuva/auth', {
      ...options,
      adapter: 'custom-session',
      provider: 'custom-session',
    })

    addPlugin({
      src: resolver.resolve('./runtime/plugin'),
    })
  },
})
```

模块仍然应该使用 `nuva.config.ts` 的 `auth` 配置，不要新增一套平行配置入口。这样业务代码始终只消费 `useAuth()` 和 `usePermission()`。

## Server adapter

如果 provider 还需要给 server utils 提供认证上下文，可以同时注册 server adapter。它和客户端 adapter 是独立 contract，适合后续接入企业 SSO、Auth.js server session 或自研 server session。

```ts
import {
  defineServerAuthAdapter,
  registerServerAuthAdapter,
} from '@oevery/nuva/auth/server-adapter'

registerServerAuthAdapter('custom-session', defineServerAuthAdapter(() => ({
  async requireAuth(event) {
    const session = await getSessionFromEvent(event)

    if (!session) {
      throw createError({ statusCode: 401, statusMessage: 'Unauthorized' })
    }

    return session
  },
})))
```

server API 中使用 `requireNuvaAuthContext(event)` 读取当前 provider 的 server adapter。返回值里如果包含 `permission`，Nuva 会同步写入 server permission context，后续可直接使用 `requireNuvaPermission()`、`requireNuvaRole()` 或 `requireNuvaScope()`。

```ts
import { requireNuvaAuthContext, requireNuvaPermission } from '@oevery/nuva/server/utils/permission'

export default defineEventHandler(async (event) => {
  const session = await requireNuvaAuthContext(event)

  requireNuvaPermission(event, 'dashboard:view')

  return session
})
```

注册 server adapter 后，也可以直接使用 provider 版本的声明式 handler：

```ts
import { defineNuvaProtectedHandler } from '@oevery/nuva/server/utils/permission'

export default defineNuvaProtectedHandler({
  roles: ['admin'],
  permissions: ['dashboard:view'],
}, (event, session, permission) => {
  return {
    session,
    permission,
  }
})
```

## 测试隔离

adapter registry 是运行时单例。单测里如果手动注册 adapter，建议在 `beforeEach` 调用 `resetAuthAdapters()`：

```ts
import { resetAuthAdapters } from '@oevery/nuva/auth/adapter'

beforeEach(() => {
  resetAuthAdapters()
})
```

`resetAuthAdapters()` 只清空外部注册的 adapter，不影响内置 token adapter。
