description: 用一套统一规则处理页面保护、登录跳转和 401，而不是在每个页面各写一份判断。
---

Nuva Auth 模块提供登录态读取、登录跳转和 route middleware。你可以用它保护需要登录的页面，并在请求失败时统一处理 401。

## 推荐做法

先决定是“默认公开，少数页面受保护”，还是“默认受保护，少数页面公开”。这个决策比具体代码更重要，因为它直接影响页面元信息怎么写。

## 默认跳转行为

未登录访问受保护页面时，会跳转到登录页并携带 redirect 参数：

```txt
/login?redirect=/profile
```

登录成功后调用 `useAuth().afterLogin()` 回到来源地址。登录页路径、默认首页和 redirect 参数名都可以在 `nuvaAuth` 中配置。

## 页面保护规则

- `nuvaAuth.global: false` 时，只有 `definePageMeta({ auth: true })` 的页面会被保护。
- `nuvaAuth.global: false` 时，如果页面声明了 `roles`、`permissions` 或 `scopes`，也会自动视为受保护页面。
- `nuvaAuth.global: true` 时，页面默认受保护。
- `definePageMeta({ auth: false })` 表示公开页面，会跳过登录态保护和路由级访问控制。
- `nuvaAuth.publicRoutes` 中的路径会跳过保护，支持精确路径和 `/path/**` 前缀匹配。

## 页面权限规则

页面可以继续声明角色、功能权限和作用域：

```ts
definePageMeta({
  auth: {
    roles: ['admin'],
    permissions: ['profile:read'],
    scopes: ['departmentId'],
    roleMode: 'any',
    permissionMode: 'all',
  },
})
```

为了兼容旧写法，`roles`、`permissions`、`scopes` 仍可直接写在 PageMeta 顶层；新页面优先使用 `auth` 对象写法，扩展性更好。

默认匹配模式来自 `nuvaAuth.permission.roleMode` 和 `nuvaAuth.permission.permissionMode`。无权限时会跳转到 `nuvaAuth.permission.forbiddenPath`，页面也可以用 `forbiddenPath` 单独覆盖。

如果 `permission.source` 是 `better-auth`，页面访问控制中的 `permissions` 应优先使用 `resource:action` 格式，并通过 `canAsync()` 做动态权限判断。`roles` 和 `scopes` 可以用于当前 active member 与 active organization 的本地判断，但它们不替代 Better Auth 的动态权限校验。

在 Better Auth 动态权限模式下，`can()` 现在会先尝试基于当前角色做同步静态判断；如果当前权限点只能由 Better Auth 服务端最终确认，则会表现为 `unknown`。这种情况下局部 UI 应优先使用 `NuvaCan` 的默认 `resolve="auto"`，或直接调用 `canAsync()`。

`NuvaCan` 默认会在遇到 `unknown` 时自动补一次异步确认，因此大多数按钮不需要再显式写 `async`。只有当你想强制同步模式，或在高密度列表里主动避免额外请求时，才建议传 `resolve="sync"`。

路由守卫只负责“能不能进页面”。按钮、表格行、表单动作等局部控制应使用 `usePermission()`：

```ts
const permission = usePermission()

const canUpdateProfile = computed(() => permission.can('profile:update'))
```

如果你希望组件自己处理 Better Auth 的最终授权，直接写：

```vue
<NuvaCan permission="project:create">
  <UButton>新建项目</UButton>
</NuvaCan>
```

服务端 API 也要做真实拦截。推荐使用声明式 handler，把业务登录校验和权限要求放在同一个入口，避免漏写接口授权：

```ts
import { definePermissionHandler } from '@oevery/nuva/server/utils/permission'

export default definePermissionHandler({
  auth: requireAuth,
  permission: 'profile:update',
}, (event, user, permission) => {
  return updateProfile(event, user)
})
```

如果接口需要同时声明角色、权限和作用域，可以使用 `defineProtectedHandler()`；低阶工具 `setNuvaAuthContext()`、`requireNuvaPermission()`、`requireNuvaRole()` 和 `requireNuvaScope()` 仍可用于更细的自定义流程。

`requireNuvaPermission()`、`requireNuvaRole()`、`requireNuvaScope()` 默认不会读取本地权限兜底；如果没有先设置当前用户权限态，会直接返回 401。这样可以避免 API 在忘记认证时被本地 demo 权限误放行。

数据权限需要结合目标资源判断：

```ts
requireNuvaDataAccess(event, {
  departmentId: profile.departmentId,
  ownerId: profile.userId,
})
```

只需要检查数据范围类型时，可以使用 `requireNuvaDataAccessType(event, 'department')`。

公开页面如果只需要局部按权限显示内容，也应使用 `usePermission()` 手动判断，不要在同一个页面同时写 `auth: false` 和路由级 `roles`、`permissions` 或 `scopes`。

## 请求 401/403 处理

`useDefaultHttpRequestHooks()` 已统一处理认证相关错误：

- `401`：调用 `useAuth().logout()` 清理登录态，并跳转登录页。
- `403`：默认提示“无权限执行该操作”，不跳转；只有页面必要请求显式声明 `forbiddenBehavior: 'redirect'` 时才跳转到 `nuvaAuth.permission.forbiddenPath`。

业务模板可以在 `app/utils/http/hooks.ts` 中扩展请求头、traceId、租户和错误提示 UI：

```ts
import { useDefaultHttpRequestHooks } from '@oevery/nuva/http'

export function useHttpRequestHooks(config) {
  const defaults = useDefaultHttpRequestHooks(config, {
    notify({ message }) {
      const toast = useToast()
      toast.add({ title: message, color: 'error' })
    },
  })

  return {
    ...defaults,
    beforeRequest(method) {
      defaults.beforeRequest(method)
      method.config.headers = {
        ...(method.config.headers || {}),
        'x-tenant-id': 'tenant-demo',
      }
    },
  }
}
```

按钮、表单提交等动作接口通常保持默认 `notify` 行为：

```ts
http.Patch('/profile', data, {
  meta: {
    errorMessage: '你没有编辑资料的权限',
  },
})
```

页面首屏必要接口如果无权限就应该离开当前页面，可以显式声明跳转：

```ts
http.Get('/profile', {
  meta: {
    forbiddenBehavior: 'redirect',
  },
})
```

静默请求可以关闭提示：

```ts
http.Get('/notifications/count', {
  meta: {
    forbiddenBehavior: 'silent',
    errorSilent: true,
  },
})
```

如果你完全接管 `onSuccess` 或 `onError`，需要继续调用 `defaults.onSuccess()` / `defaults.onError()`，或者手动复用 `handleAuthHttpError()`，避免绕过统一的 401/403 行为。
