title: 路由保护与登录态
description: 用一套统一规则处理页面保护、登录跳转和 401，而不是在每个页面各写一份判断。
---

Nuva Auth 模块提供登录态读取、登录跳转和 route middleware。你可以用它保护需要登录的页面，并在请求失败时统一处理 401。

## 推荐做法

先决定是“默认公开，少数页面受保护”，还是“默认受保护，少数页面公开”。这个决策比具体代码更重要，因为它直接影响页面元信息怎么写。

## 默认跳转行为

未登录访问受保护页面时，会跳转到登录页并携带 redirect 参数：

```txt
/login?redirect=/profile
```

登录成功后调用 `useAuth().afterLogin()` 回到来源地址。登录页路径、默认首页和 redirect 参数名都可以在 `nuvaAuth` 中配置。

## 页面保护规则

- `nuvaAuth.global: false` 时，只有 `definePageMeta({ auth: true })` 的页面会被保护。
- `nuvaAuth.global: false` 时，如果页面在 `auth` 对象里声明了 `roles`、`permissions` 或 `scopes`，也会自动视为受保护页面。
- `nuvaAuth.global: true` 时，页面默认受保护。
- `definePageMeta({ auth: false })` 表示公开页面，会跳过登录态保护和路由级访问控制。
- `nuvaAuth.publicRoutes` 中的路径会跳过保护，支持精确路径和 `/path/**` 前缀匹配。

如果路径同时命中 `publicRoutes`，并且页面显式声明了 `auth: true` 或 `auth: { ... }`，页面元信息优先，仍会走登录和权限保护。开发环境会输出 warning，提示清理冲突配置。

## 页面权限规则

页面可以继续声明角色、功能权限和作用域：

```ts
definePageMeta({
  auth: {
    roles: ['admin'],
    permissions: ['profile:read'],
    scopes: ['departmentId'],
    roleMode: 'any',
    permissionMode: 'all',
  },
})
```

默认匹配模式来自 `nuvaAuth.permission.roleMode` 和 `nuvaAuth.permission.permissionMode`。无权限时会跳转到 `nuvaAuth.permission.forbiddenPath`，页面也可以用 `forbiddenPath` 单独覆盖。

如果 `permission.source` 是 `adapter` 且当前 provider 是 Better Auth，页面访问控制中的 `permissions` 应优先使用 `resource:action` 格式，并通过 `canAsync()` 做动态权限判断。`roles` 和 `scopes` 可以用于当前 active member 与 active organization 的本地判断，但它们不替代 Better Auth 的动态权限校验。

在 Better Auth 动态权限模式下，`can()` 现在会先尝试基于当前角色做同步静态判断；如果当前权限点只能由 Better Auth 服务端最终确认，则会表现为 `unknown`。这种情况下局部 UI 应优先使用 `NuvaCan` 的默认 `resolve="auto"`，或直接调用 `canAsync()`。

`NuvaCan` 默认会在遇到 `unknown` 时自动补一次异步确认，并在 pending 期间隐藏内容，避免按钮先显示“无权限”再闪烁成可用。只有当你想强制同步模式，或在高密度列表里主动避免额外请求时，才建议传 `resolve="sync"`。

路由守卫只负责“能不能进页面”。按钮、表格行、表单动作等局部控制应使用 `usePermission()`：

```ts
const permission = usePermission()

const canUpdateProfile = computed(() => permission.can('profile:update'))
```

如果你希望组件自己处理 Better Auth 的最终授权，直接写：

```vue
<NuvaCan permission="project:create">
  <UButton>新建项目</UButton>
</NuvaCan>
```

如果希望异步确认期间显示骨架或禁用态，可以使用 `pending` slot：

```vue
<NuvaCan permission="project:create">
  <UButton>新建项目</UButton>

  <template #pending>
    <UButton disabled>
      权限确认中
    </UButton>
  </template>

  <template #fallback>
    <UButton disabled>
      无权限
    </UButton>
  </template>
</NuvaCan>
```

如果需要旧行为，也就是 pending 期间直接显示 fallback，可以显式设置 `pendingBehavior="fallback"`。

服务端 API 也要做真实拦截。推荐使用声明式 handler，把业务登录校验和权限要求放在同一个入口，避免漏写接口授权。多数项目优先使用当前 provider 的 server adapter 版本：

```ts
import { defineNuvaPermissionHandler } from '@oevery/nuva/server/utils/permission'

export default defineNuvaPermissionHandler({
  permission: 'profile:update',
}, (event, auth, permission) => {
  return updateProfile(event, auth)
})
```

Better Auth 或自定义 adapter 的动态权限可以携带服务端上下文，最终授权仍由 provider 决定：

```ts
import { defineNuvaPermissionHandler } from '@oevery/nuva/server/utils/permission'

export default defineNuvaPermissionHandler({
  permission: 'project:update',
  context: event => ({
    target: {
      projectId: getRouterParam(event, 'id'),
    },
  }),
}, (event, auth, permission) => {
  return updateProject(event, auth)
})
```

`@oevery/nuva/better-auth` 会自动注册 Better Auth server adapter；自定义 session、Auth.js 或企业 SSO 可以通过 [Auth adapter 开发](/auth/adapters) 中的 server adapter contract 注册。

server adapter 的 `requireAuth()` 如果返回 `null` 或 `undefined`，Nuva 会强制按未登录处理并返回 `401`，避免 adapter 实现不严谨时误放行。

如果接口需要同时声明角色、权限和作用域，优先使用 `defineNuvaProtectedHandler()`；需要自己传入 `auth` 函数时再使用 `defineProtectedHandler()`。

低阶工具 `setNuvaAuthContext()`、`requireNuvaPermission()`、`requireNuvaRole()` 和 `requireNuvaScope()` 只建议放在高度自定义流程里，不作为普通 API 的默认入口。

`requireNuvaPermission()`、`requireNuvaRole()`、`requireNuvaScope()` 默认不会读取本地权限兜底；如果没有先设置当前用户权限态，会直接返回 401。这样可以避免 API 在忘记认证时被本地 demo 权限误放行。

数据权限需要结合目标资源判断：

```ts
requireNuvaDataAccess(event, {
  departmentId: profile.departmentId,
  ownerId: profile.userId,
})
```

只需要检查数据范围类型时，可以使用 `requireNuvaDataAccessType(event, 'department')`。

公开页面如果只需要局部按权限显示内容，也应使用 `usePermission()` 手动判断，不要在同一个页面同时写 `auth: false` 和路由级 `auth.roles`、`auth.permissions` 或 `auth.scopes`。

## 请求 401/403 处理

`useDefaultHttpRequestHooks()` 已统一处理认证相关错误：

- `401`：调用 `useAuth().logout()` 清理登录态，并跳转登录页。
- `403`：默认提示“无权限执行该操作”，不跳转；只有页面必要请求显式声明 `forbiddenBehavior: 'redirect'` 时才跳转到 `nuvaAuth.permission.forbiddenPath`。

业务模板可以在 `app/utils/http/hooks.ts` 中扩展请求头、traceId、租户和错误提示 UI：

```ts
import { useDefaultHttpRequestHooks } from '@oevery/nuva/http'

export function useHttpRequestHooks(config) {
  const defaults = useDefaultHttpRequestHooks(config, {
    notify({ message }) {
      const toast = useToast()
      toast.add({ title: message, color: 'error' })
    },
  })

  return {
    ...defaults,
    beforeRequest(method) {
      defaults.beforeRequest(method)
      method.config.headers = {
        ...(method.config.headers || {}),
        'x-tenant-id': 'tenant-demo',
      }
    },
  }
}
```

按钮、表单提交等动作接口通常保持默认 `notify` 行为：

```ts
http.Patch('/profile', data, {
  meta: {
    errorMessage: '你没有编辑资料的权限',
  },
})
```

页面首屏必要接口如果无权限就应该离开当前页面，可以显式声明跳转：

```ts
http.Get('/profile', {
  meta: {
    forbiddenBehavior: 'redirect',
  },
})
```

静默请求可以关闭提示：

```ts
http.Get('/notifications/count', {
  meta: {
    forbiddenBehavior: 'silent',
    errorSilent: true,
  },
})
```

如果你完全接管 `onSuccess` 或 `onError`，需要继续调用 `defaults.onSuccess()` / `defaults.onError()`，或者手动复用 `handleAuthHttpError()`，避免绕过统一的 401/403 行为。

菜单来自远程接口时，可以在测试或 CI 中复用 Nuva 的一致性校验，避免菜单权限比路由权限更宽：

```ts
import { validateAccessMenus } from '@oevery/nuva/auth/access-menu'

const issues = validateAccessMenus(menus, router.getRoutes(), {
  routePrune: true,
  strictRoute: true,
})
```
