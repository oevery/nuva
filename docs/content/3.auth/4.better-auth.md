---

`fullstack` 模式会把 Better Auth 挂载到 `runtimeConfig.public.nuva.auth.betterAuth.basePath`，默认是 `/api/auth`。

这一步通常发生在你已经确认项目需要完整认证协议，而不是只做 token 登录态管理之后。

## 启用方式

业务项目需要提供 `nuvaAuth.betterAuth.serverAuthImport` 指向的 auth 实例，默认路径是 `~~/server/utils/better-auth`。

Nuva 只负责把 Better Auth 挂到 Nuxt 并提供最小桥接。认证方式、organization、teams、admin、custom permissions 这些能力本身仍应按 Better Auth 文档配置，而不是在 Nuva 里再封一层。

```ts
export default defineNuxtConfig({
  modules: ['@oevery/nuva/auth'],
})
```

```ts
// nuva.config.ts
import { defineNuvaConfig } from '@oevery/nuva/config'

export default defineNuvaConfig({
  auth: {
    preset: 'better-auth',
    mode: 'fullstack',
    provider: 'better-auth',
    permission: {
      provider: 'better-auth',
      betterAuth: {
        hasPermission: true,
      },
    },
    betterAuth: {
      basePath: '/api/auth',
      serverAuthImport: '~~/server/utils/better-auth',
    },
  },
})
```

## 路径约定

- `/api/auth/**` 保留给 Better Auth handler。
- token/cookie demo 登录接口建议使用 `/api/demo-auth/*`，避免与正式协议冲突。模板里的 demo auth 仅用于演示本地权限链路，默认使用可配置凭据和签名 token；生产环境应替换为真实认证，或至少显式设置 `NUVA_DEMO_AUTH_SECRET`。

## 生产安全基线

模板里的 `server/utils/better-auth.ts` 已显式配置一组保守默认值：

- `secret`：生产环境必须设置 `BETTER_AUTH_SECRET` 或 `AUTH_SECRET`，且至少 32 个字符。
- `trustedOrigins`：通过 `BETTER_AUTH_TRUSTED_ORIGINS` 配置允许的前端来源，多个来源用英文逗号分隔。
- `baseURL`：通过 `BETTER_AUTH_URL` 或 `BETTER_AUTH_BASE_URL` 指定正式服务地址。
- `rateLimit`：默认开启全局限流，并对邮箱登录、注册端点使用更严格规则。
- `session`：默认 7 天过期，24 小时更新一次，会话 cookie cache 默认 5 分钟。
- `CSRF`：显式保持 Better Auth CSRF 校验开启，不要在生产环境关闭。
- `cookie`：生产环境启用 secure cookie，默认 `sameSite: 'lax'`、`path: '/'`。

推荐生产环境变量：

```bash
BETTER_AUTH_SECRET="$(openssl rand -base64 32)"
BETTER_AUTH_URL="https://app.example.com"
BETTER_AUTH_TRUSTED_ORIGINS="https://app.example.com,https://admin.example.com"
```

如果保留模板 demo auth，至少也要设置：

```bash
NUVA_DEMO_AUTH_SECRET="$(openssl rand -base64 32)"
NUVA_DEMO_AUTH_USERNAME="admin"
NUVA_DEMO_AUTH_PASSWORD="replace-with-a-strong-password"
```

demo auth 只适合本地开发和权限链路演示。它虽然包含 HMAC 签名 token、httpOnly cookie、过期时间、生产强 secret 校验和基础登录限流，但不具备服务端 session 撤销、refresh token、账号锁定、审计日志等生产认证能力。

上线前至少确认：

- 使用 Better Auth 或企业统一认证替换 demo auth。
- 生产环境启用 HTTPS。
- 不提交任何 secret 到代码仓库。
- 正确配置 `BETTER_AUTH_TRUSTED_ORIGINS`，不要使用过宽泛通配符。
- 如果在反向代理后运行，明确配置真实 IP 头并只信任可信代理。
- 对登录、注册、重置密码等敏感接口保留限流。
- OAuth 场景如需存储第三方 token，开启 Better Auth 的 OAuth token 加密能力。

## 客户端调用方式

Better Auth 协议请求使用官方 client，不走 alova：

```ts
const { betterAuthClient } = useAuth()
const authClient = betterAuthClient || useBetterAuth()
const { data: session } = authClient.useSession()
```

当 `preset: 'better-auth'` 或 `provider: 'better-auth'` 启用时，`useAuth().isAuthenticated`、`useAuth().ready`、`useAuth().user` 和 `useAuth().logout()` 会基于 Better Auth session，而不是 token 状态。

如果你在 SSR 页面里调用 Better Auth 除 `useSession(useFetch)` 之外的接口，Nuva 会按 Better Auth 的 Nuxt 文档创建 request-scoped client 并自动转发 `cookie`。不要再为 `/api/auth/**` 额外写 alova wrapper。

例如邮箱登录：

```ts
await authClient.signIn.email({
  email,
  password,
})
```

## 和业务 API 的边界

- 业务 API 走 alova 和 `useHttpClient()`。
- Better Auth 的 `/api/auth/**` 走 `useBetterAuth()`。

不要再为 `/api/auth/**` 额外写 alova wrapper；这组端点由 Better Auth client 管理。

如果你的项目里同时存在业务接口和 Better Auth，最重要的不是“统一成一套调用方式”，而是边界清晰，避免认证协议和业务请求互相污染。

## 和权限系统的边界

Better Auth organization 的 access control 和 dynamic access control 可以作为 Nuva 的权限来源，但不要把 organization、member、team 等对象直接暴露给业务页面。

推荐做法是：

- Better Auth 负责认证协议、组织插件和动态权限校验。
- Nuva 在非 Better Auth 模式下继续负责通用权限抽象；在 Better Auth 模式下只做薄适配，不再兼容额外的 `getInfo/profile` 权限协议。
- 业务页面只使用 `usePermission()`。

```ts
const permission = usePermission()

await permission.canAsync('project:create')
```

`can()` 使用当前快照做同步判断。对于 Better Auth，它会优先基于 `activeMember.role` 和 `organization.checkRolePermission()` 做静态角色权限判断；如果当前权限点需要 Better Auth 服务端最终确认，则该判断会退化为 `unknown`。`canAsync()` 在 `permission.provider: 'better-auth'` 且 `permission.betterAuth.hasPermission: true` 时会调用 Better Auth organization `hasPermission`，适合动态角色或需要服务端确认的权限点。

NuvaCan 默认会在同步判断为 `unknown` 时自动补一次 Better Auth 最终授权，并在 pending 期间隐藏内容，避免按钮先显示 fallback 再闪烁。因此大多数局部按钮可以直接这样写：

```vue
<NuvaCan permission="project:create">
  <UButton>新建项目</UButton>
</NuvaCan>
```

如果需要 pending 期间展示骨架或禁用按钮，可以使用 `pending` slot：

```vue
<NuvaCan permission="project:create">
  <UButton>新建项目</UButton>

  <template #pending>
    <UButton disabled>
      权限确认中
    </UButton>
  </template>

  <template #fallback>
    <UButton disabled>
      无权限
    </UButton>
  </template>
</NuvaCan>
```

如果你需要 pending 期间立即显示 fallback，可以设置 `pendingBehavior="fallback"`。

如果你想强制同步或强制异步，也可以显式指定：

```vue
<NuvaCan resolve="sync" permission="project:create">
  <UButton>只做本地静态判断</UButton>
</NuvaCan>

<NuvaCan resolve="async" permission="project:create">
  <UButton>总是走最终授权</UButton>
</NuvaCan>
```

Nuva 只会在配置开启后自动给 Better Auth client 挂上 `organizationClient()`。服务端仍需要你在 Better Auth 实例中启用 `organization()` 插件，并按 Better Auth 的要求配置 access control、roles 或 dynamic access control。

如果你只需要 Better Auth session，不需要 organization 权限，可以保留 `provider: 'better-auth'`，但不要开启 `permission.provider: 'better-auth'` 或 `permission.betterAuth.hasPermission`。

在 Better Auth 模式下，`can()` 只代表“本地同步静态判断”；真正的动态授权仍应使用 `canAsync()`。不过 `NuvaCan` 已经把这层自动补齐，适合按钮、菜单和局部 UI。`hasRole()` 和 `hasScope()` 会基于当前 active member / active organization 提供本地判断，适合页面骨架、组织级 UI 切换和非敏感展示场景。

Better Auth 的 `hasPermission` 需要结构化权限，因此 Nuva 默认只把 `resource:action` 格式转换为 Better Auth 权限对象，例如 `project:create`。如果你的项目使用其它命名格式，需要先映射成 `resource:action`，不要依赖本地缓存兜底。

页面 meta 也遵循同样规则：Better Auth 模式下 `permissions` 优先用于动态权限保护；`roles` 与 `scopes` 更适合表达当前活跃组织成员角色与组织作用域要求。
