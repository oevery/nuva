---
title: Auth 总览
description: 用最小配置启用 Nuva Auth，并明确 Auth core 与 Better Auth adapter 的职责边界。
---

Auth 是 Nuva 的可选能力。项目通过 `extends: ['@oevery/nuva']` 使用 Nuva 基础层后，会自动读取项目根目录的 `nuva.config.ts`；启用 `@oevery/nuva/auth` 时，auth core 只消费这份统一配置并补充登录态、跳转、权限和路由保护。如果你还没有决定认证方案，先从 token/remote 模式开始更稳妥。

## 启用模块

已有后端登录接口时使用 Auth core：

```ts
export default defineNuxtConfig({
  modules: ['@oevery/nuva/auth'],
})
```

```ts
// nuva.config.ts
import { defineNuvaConfig } from '@oevery/nuva/config'

export default defineNuvaConfig({
  auth: {
    provider: 'token',
    loginPath: '/login',
    homePath: '/',
    global: true,
    publicRoutes: ['/login'],
    permission: {
      source: 'remote',
    },
  },
})
```

## Provider

- `@oevery/nuva/auth`：提供 `useAuth()`、登录跳转、权限和 route middleware，登录接口由项目自己实现。
- `@oevery/nuva/better-auth`：自动挂载 Auth core，并额外注册 Better Auth client、session adapter 与 `/api/auth/**` handler。

配置只描述 provider，不再使用 `preset` 或 `mode`：

- `auth.provider`：谁负责登录态，例如 `token`、`better-auth` 或自定义 adapter。
- `permission.source`：权限来源，例如 `local`、`remote`、`hybrid`、`adapter`。
- `accessMenu.source`：菜单来源，例如 `none`、`profile`、`remote`、`route`。

多数国内后台优先从 `remote` 开始：登录接口返回 token，`remote.profile` 返回用户、角色、按钮权限和数据权限，等同常见的 `/getInfo` 模式。

如果只想快速复制配置，可以先看 [认证与权限](/auth) 里的动态接入示例。

## 什么时候选 Auth core

- 你已有后端登录接口。
- 你只想统一 token、跳转和页面保护。
- 你不想让 `/api/auth/**` 成为项目里的核心协议入口。

## 什么时候选 Better Auth

- 你准备使用 Better Auth 管理认证协议。
- 你希望认证逻辑和 Nuxt server API 放在一起维护。
- 你可以接受业务 API 与认证 API 分开走两套客户端入口。

```ts
export default defineNuxtConfig({
  modules: ['@oevery/nuva/better-auth'],
})
```

```ts
// nuva.config.ts
import { defineNuvaConfig } from '@oevery/nuva/config'

export default defineNuvaConfig({
  auth: {
    provider: 'better-auth',
  },
})
```

## 推荐入口

日常业务只需要记住四个入口：

- 登录态和跳转：`useAuth()`。
- 权限判断：`usePermission()`。
- 局部 UI 显示：`NuvaCan`。
- 服务端 API 授权：`defineNuvaPermissionHandler()`。

应用层读取登录态优先使用 `useAuth()`：

```ts
const auth = useAuth()

if (!auth.isAuthenticated.value) {
  await auth.toLogin()
}
```

如果你只是读取最终 Nuva 配置，客户端和通用运行时优先使用 `useNuvaConfig()`：

```ts
const nuva = useNuvaConfig()
const authConfig = nuva.auth
const apiConfig = nuva.api
```

服务端 `server/api` 或 `server/utils` 中已有 `event` 时，优先使用 `getNuvaConfig(event)`，避免依赖组合式上下文：

```ts
export default defineEventHandler((event) => {
  const nuva = getNuvaConfig(event)
  return nuva.auth.loginPath
})
```

## `useAuth()` 提供什么

- `provider`
- `user`
- `ready`
- `isAuthenticated`
- `refresh`
- `logout`
- `toLogin`
- `afterLogin`

## 权限模型

Nuva 把权限拆成两层，不直接绑定 Better Auth 的组织模型：

- 功能权限：`permissions`，推荐 `resource:action` 格式，例如 `profile:update`。
- 作用域和数据范围：`scope` 与 `dataAccess`，用于表达用户、租户、组织、部门、门店、项目或自定义数据范围。

这样做的好处是：简单项目只用 `permissions` 就够；复杂项目可以继续接入租户、部门、门店、组织或 Better Auth organization，但业务侧仍然只消费统一模型。

```ts
const permission = usePermission()

permission.can('profile:update')
await permission.canAsync('profile:update')
permission.any(['profile:update', 'profile:approve'])
permission.hasRole('admin')
permission.hasScope('departmentId')
```

应用代码主入口保持简单：单个权限点用 `can()` / `canAsync()`，多个权限组合用 `any()` / `all()` / `anyAsync()` / `allAsync()`。`canState()`、`anyState()`、`allState()` 和 `explain()` 属于诊断或高级 UI 场景，用来区分 `allow`、`deny`、`unknown`，不要作为普通业务页的默认入口。

大型项目可以用 `defineNuvaPermissions()` 集中维护权限字典。它不会改变运行时字符串模型，只是让业务代码拿到更稳定的类型约束：

```ts
import { defineNuvaPermissions } from '@oevery/nuva/auth/permission'

export const permissions = defineNuvaPermissions({
  profileRead: 'profile:read',
  profileUpdate: 'profile:update',
})

permission.can(permissions.profileUpdate)
```

调试权限问题时，可以使用 `explain()`、`explainRole()` 和 `explainScope()` 查看缺失项：

```ts
const result = permission.explain(['profile:read', 'profile:update'])
// { decision, reason, required, matched, missing, mode }
```

模板里可以用 `NuvaCan` 做局部权限显示：

```vue
<NuvaCan permission="profile:update">
  <UButton>保存</UButton>
</NuvaCan>
```

默认情况下，`NuvaCan` 会先走同步判断；如果当前权限来源是 adapter 且只能异步确认，组件会自动补一次最终授权。只有在你明确要限制行为时，才需要传 `resolve="sync"` 或 `resolve="async"`。

登录接口返回 token 和用户态时，使用 token provider client 同步 token、user 和权限缓存：

```ts
const tokenAuth = useTokenAuthClient<CurrentUser>()

tokenAuth.setSession(data.token, data.user)
```

`useTokenAuthClient()` 是 token provider 专用 API，只在你的登录接口返回 token 时使用。业务页面读取登录态仍然只使用 `useAuth()`。

`usePermission().ready` 和 `usePermission().permissionReady` 都表示权限态已经可用；`local` 和 `hybrid` 模式会立即有本地权限态，`remote` 模式需要从用户态或权限接口拿到权限后才会 ready。如果你需要判断认证状态是否初始化，使用 `usePermission().authReady` 或 `useAuth().ready`。

`useAuth().logout()` 会清理登录态相关的权限与菜单缓存。切换租户、组织或账号上下文时，如果你接管了切换流程，也可以显式调用：

```ts
import { clearNuvaAuthDerivedState } from '@oevery/nuva/auth/state'

clearNuvaAuthDerivedState()
```

## 权限来源

`nuvaAuth.permission.source` 支持四种来源：

- `local`：本地静态角色和权限。
- `remote`：后端接口返回用户角色、权限、作用域和数据范围。
- `hybrid`：本地提供保底权限，远程接口刷新真实权限。
- `adapter`：由当前 auth provider 的 adapter 提供权限态和动态权限校验。Better Auth organization/access control 就走这条路径。

Better Auth 是可选 adapter module，不是 Nuva 权限系统本身。即使使用 Better Auth，业务代码也应优先使用 `usePermission()`；启用 `betterAuth.organization.*` 后，`permission.source` 会被设置为 `adapter`，不再混用额外的 `getInfo` 风格接口。

## 渐进增强路径

Auth 不要求一开始就接完整权限系统，可以按项目成熟度逐步增加配置：

| 阶段 | 配置重点 | 适用场景 |
| --- | --- | --- |
| Token only | `auth.provider: 'token'` | 先统一登录态、跳转和路由保护。 |
| Local | `permission.source: 'local'` | Demo、静态原型、开发期兜底。 |
| Remote profile | `permission.source: 'remote'` + `remote.profile` | 常见 `/getInfo`，一次返回用户、角色、权限和菜单。 |
| Remote permission | `remote.permission` | 权限服务独立维护或需要单独刷新权限。 |
| Hybrid | `permission.source: 'hybrid'` | 本地兜底首屏，远程刷新真实权限。 |
| Adapter | `permission.source: 'adapter'` | Better Auth 或自定义认证系统接管权限判断。 |

这条路径是叠加式的：简单项目可以停在 token/remote，复杂项目再接 Better Auth organization 或自定义 server adapter。

Core 默认使用 `local`，不会请求模板里的 demo 接口。使用 `remote` 或 `hybrid` 时，需要显式提供远程权限来源：

```ts
import { defineNuvaConfig } from '@oevery/nuva/config'

export default defineNuvaConfig({
  auth: {
    provider: 'token',
    permission: {
      source: 'remote',
      remote: {
        profile: {
          url: '/auth/me',
          method: 'GET',
        },
        cacheMaxAge: 0,
      },
    },
  },
})
```

`cacheMaxAge` 控制路由守卫中用户态和权限态的缓存有效期。默认值是 `0`，表示每次守卫都会重新校验；只有确认可以接受撤权延迟时，才建议显式配置缓存时间。

Better Auth provider 下，`usePermission().ready` 表示 Better Auth session 与组织上下文已经完成初始化；真正的动态权限判断仍应使用 `canAsync()`。如果你需要区分“明确拒绝”和“当前只能异步确认”，可以使用 `canState()`，它会返回 `allow`、`deny` 或 `unknown`。

远程权限协议只看 `remote.profile` 和 `remote.permission` 是否配置：`remote.profile` 适合 `/getInfo` 一次返回用户、角色、权限、作用域和数据范围；`remote.permission` 适合权限服务独立维护。两者都配置时，用户态从 `profile` 刷新，权限态可由 `permission` 单独刷新。`hybrid` 在远程权限不可用前会继续使用本地权限作为保底。复杂场景下建议把接入逻辑放到 `app/nuva/profile-resolver.ts` 或 `app/nuva/permission-resolver.ts`，Nuva 会按约定自动发现；如果使用自定义路径，再在 `nuva.config.ts` 的 `resolvers.profile` 或 `resolvers.permission` 中显式指定。
