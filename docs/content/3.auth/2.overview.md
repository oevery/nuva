Auth 是 Nuva 的可选能力。项目通过 `extends: ['@oevery/nuva']` 使用 Nuva 基础层后，会自动读取项目根目录的 `nuva.config.ts`；启用 `@oevery/nuva/auth` 时，auth 模块只消费这份统一配置并补充认证能力。如果你还没有决定认证方案，先从 `frontend` 模式开始更稳妥。

## 启用模块

```ts
export default defineNuxtConfig({
  modules: ['@oevery/nuva/auth'],
})
```

```ts
// nuva.config.ts
import { defineNuvaConfig } from '@oevery/nuva/config'

export default defineNuvaConfig({
  auth: {
    preset: 'remote',
    mode: 'frontend',
    loginPath: '/login',
    homePath: '/',
    global: true,
    publicRoutes: ['/login'],
  },
})
```

## 模式

- `frontend`：提供 `useAuth()`、登录跳转和 route middleware，登录接口由项目自己实现。
- `fullstack`：在 `frontend` 能力基础上挂载 Better Auth handler。

`preset` 是快速入口，显式配置仍优先生效：

- `token`：本地 token 登录态，权限默认 local。
- `remote`：token 登录态，用户态或权限接口下发权限。
- `hybrid`：本地权限先可用，远程权限刷新真实授权。
- `better-auth`：Better Auth session + organization/access control；这条路径不再复用 `remote.profile` 或 `remote.permission` 协议。

多数国内后台优先从 `remote` 开始：登录接口返回 token，`remote.profile` 返回用户、角色、按钮权限和数据权限，等同常见的 `/getInfo` 模式。

如果只想快速复制配置，可以先看 [认证与权限](/auth) 里的动态接入示例。

## 什么时候选 `frontend`

- 你已有后端登录接口。
- 你只想统一 token、跳转和页面保护。
- 你不想让 `/api/auth/**` 成为项目里的核心协议入口。

## 什么时候选 `fullstack`

- 你准备使用 Better Auth 管理认证协议。
- 你希望认证逻辑和 Nuxt server API 放在一起维护。
- 你可以接受业务 API 与认证 API 分开走两套客户端入口。

## 推荐入口

应用层优先使用 `useAuth()`：

```ts
const auth = useAuth()

if (!auth.isAuthenticated.value) {
  await auth.toLogin()
}
```

如果你只是读取最终 Nuva 配置，客户端和通用运行时优先使用 `useNuvaConfig()`：

```ts
const nuva = useNuvaConfig()
const authConfig = nuva.auth
const apiConfig = nuva.api
```

服务端 `server/api` 或 `server/utils` 中已有 `event` 时，优先使用 `getNuvaConfig(event)`，避免依赖组合式上下文：

```ts
export default defineEventHandler((event) => {
  const nuva = getNuvaConfig(event)
  return nuva.auth.loginPath
})
```

## `useAuth()` 提供什么

- `mode`
- `user`
- `ready`
- `isAuthenticated`
- `logout`
- `loginWithToken`
- `toLogin`
- `afterLogin`
- `permission`

## 权限模型

Nuva 把权限拆成两层，不直接绑定 Better Auth 的组织模型：

- 功能权限：`permissions`，推荐 `resource:action` 格式，例如 `profile:update`。
- 作用域和数据范围：`scope` 与 `dataAccess`，用于表达用户、租户、组织、部门、门店、项目或自定义数据范围。

这样做的好处是：简单项目只用 `permissions` 就够；复杂项目可以继续接入租户、部门、门店、组织或 Better Auth organization，但业务侧仍然只消费统一模型。

```ts
const permission = usePermission()

permission.can('profile:update')
permission.canState('profile:update')
permission.any(['profile:update', 'profile:approve'])
await permission.anyAsync(['profile:update', 'profile:approve'])
permission.hasRole('admin')
permission.hasScope('departmentId')
```

`can()` / `canState()` / `canAsync()` 推荐只用于单个权限点；多个权限组合时，优先使用 `any()`、`all()`、`anyState()`、`allState()`、`anyAsync()` 或 `allAsync()`，避免把组合语义隐式塞进 `can()`。

模板里可以用 `NuvaCan` 做局部权限显示：

```vue
<NuvaCan permission="profile:update">
  <UButton>保存</UButton>
</NuvaCan>
```

默认情况下，`NuvaCan` 会先走同步判断；如果当前权限来源是 Better Auth 且只能异步确认，组件会自动补一次最终授权。只有在你明确要限制行为时，才需要传 `resolve="sync"` 或 `resolve="async"`。

登录接口返回 token 和用户态时，优先用 `loginWithToken()` 同步 token、user 和权限缓存：

```ts
const auth = useAuth<CurrentUser>()

auth.loginWithToken(data.token, data.user)
```

`usePermission().ready` 和 `usePermission().permissionReady` 都表示权限态已经可用；`local` 和 `hybrid` 模式会立即有本地权限态，`remote` 模式需要从用户态或权限接口拿到权限后才会 ready。如果你需要判断认证状态是否初始化，使用 `usePermission().authReady` 或 `useAuth().ready`。

## 权限来源

`nuvaAuth.permission.source` 支持四种来源：

- `local`：本地静态角色和权限。
- `remote`：后端接口直接返回用户角色、权限、作用域和数据范围。
- `hybrid`：本地提供保底权限，远程接口刷新真实权限。
- `better-auth`：用 Better Auth organization/access control 做动态权限校验，需要显式开启 `permission.betterAuth.hasPermission`。

Better Auth 是可选适配器，不是 Nuva 权限系统本身。即使使用 fullstack 模式，业务代码也应优先使用 `usePermission()`；但在 Better Auth 模式下，权限来源就是 Better Auth 自己的 organization/access control，不再混用额外的 `getInfo` 风格接口。

Core 默认使用 `local`，不会请求模板里的 demo 接口。使用 `remote` 或 `hybrid` 时，需要显式提供远程权限来源：

```ts
import { defineNuvaConfig } from '@oevery/nuva/config'

export default defineNuvaConfig({
  auth: {
    preset: 'remote',
    permission: {
      provider: 'profile',
      remote: {
        profile: {
          url: '/auth/me',
          method: 'GET',
        },
        cacheMaxAge: 0,
      },
    },
  },
})
```

`cacheMaxAge` 控制路由守卫中用户态和权限态的缓存有效期。默认值是 `0`，表示每次守卫都会重新校验；只有确认可以接受撤权延迟时，才建议显式配置缓存时间。

Better Auth 模式下，`usePermission().ready` 表示 Better Auth session 与组织上下文已经完成初始化；真正的动态权限判断仍应使用 `canAsync()`。如果你需要区分“明确拒绝”和“当前只能异步确认”，可以使用 `canState()`，它会返回 `allow`、`deny` 或 `unknown`。

远程权限协议有两种推荐写法：`provider: 'profile'` 让 `remote.profile` 直接返回 `roles`、`permissions`、`scope`、`dataAccess`；`provider: 'endpoint'` 则只从 `remote.permission` 读取权限。`remote` 允许两者组合，`hybrid` 在远程权限不可用时会继续使用本地权限作为保底。复杂场景下建议把接入逻辑放到 `app/nuva/profile-resolver.ts` 或 `app/nuva/permission-resolver.ts`，Nuva 会按约定自动发现；如果使用自定义路径，再在 `nuva.config.ts` 的 `resolvers.profile` 或 `resolvers.permission` 中显式指定。
