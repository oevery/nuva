description: 理解 alova、useHttpClient、响应解包、成功码和请求 hooks 在 Nuva 中的职责。
---

Nuva 模板使用 alova 作为业务请求入口。核心约定是：请求基础能力统一配置一次，业务接口按领域组织，页面只消费 composable 返回的状态。

## 请求层解决什么

- 统一 API 基础地址。
- 统一 token 注入。
- 统一响应类型解析。
- 统一 `{ code, message, data }` 解包。
- 统一业务成功码。
- 统一 401/403 和错误提示。

## 配置入口

请求配置最终位于 `runtimeConfig.public.nuva.api`，通常通过 `nuva.config.ts` 维护：

```ts
import { defineNuvaConfig } from '@oevery/nuva/config'

export default defineNuvaConfig({
  api: {
    baseURL: '/api',
    envelopeUnwrap: true,
    successCodes: '0,200,SUCCESS',
  },
})
```

## 响应协议

常见中后台接口会返回统一包裹结构：

```ts
export interface ApiResponse<T = unknown> {
  code: number
  message: string
  data: T
}
```

当 `envelopeUnwrap` 为 `true` 时，`useHttpClient()` 会校验 `code`，成功后返回 `data`。页面不需要反复写 `response.data.data`。

## 业务 API 组织方式

推荐把业务请求放在：

```txt
app/composables/apis/*.ts
```

同一业务域尽量收敛到一个文件：

- `useXxxApi()` 放接口方法。
- `useXxxTable()` 放分页、筛选等状态组合。
- 页面只负责调用，不直接处理 baseURL、token、响应解包和错误协议。

## 单接口覆盖

单个接口可以通过 alova `meta` 覆盖全局行为：

```ts
http.Get('/profile', {
  meta: {
    ignoreToken: true,
    responseType: 'json',
    envelopeUnwrap: true,
    successCodes: '0,200',
  },
})
```

完整 meta 字段见 [HTTP API 参考](/reference/http)。

## 错误处理边界

普通接口错误在请求 hooks 中处理；登录态失效和 401/403 的行为由 Auth 配置和 `useDefaultHttpRequestHooks()` 统一处理。

模板可以在 `app/utils/http/hooks.ts` 中追加请求头、traceId、租户和错误提示 UI。
