Nuva 模板使用 alova 作为业务请求入口。核心约定是：请求基础能力统一配置一次，业务接口按领域组织，页面只消费 composable 返回的状态。

## 请求层解决什么

- 统一 API 基础地址。
- 统一 token 注入。
- 统一响应类型解析。
- 统一 `{ code, message, data }` 解包。
- 统一业务成功码。
- 统一 401/403 和错误提示。
- 提供分页和资源型 CRUD 的轻量 helper。

## 配置入口

请求配置最终位于 `runtimeConfig.public.nuva.api`，通常通过 `nuva.config.ts` 维护：

```ts
import { defineNuvaConfig } from '@oevery/nuva/config'

export default defineNuvaConfig({
  api: {
    baseURL: '/api',
    envelopeUnwrap: true,
    successCodes: '0,200,SUCCESS',
    response: {
      codeKey: 'code',
      messageKey: 'message',
      dataKey: 'data',
    },
    pagination: {
      pageField: 'pageNum',
      pageSizeField: 'pageSize',
      listKey: 'list',
      totalKey: 'total',
    },
  },
})
```

## 响应协议

常见中后台接口会返回统一包裹结构：

```ts
export interface ApiResponse<T = unknown> {
  code: number
  message: string
  data: T
}
```

当 `envelopeUnwrap` 为 `true` 时，`useHttpClient()` 会按 `api.response` 校验业务码，成功后返回 `dataKey` 对应数据。页面不需要反复写 `response.data.data`。

Nuva 会在 `nuxt prepare/dev/build/typecheck` 阶段根据静态 API 协议生成当前项目类型：

```ts
import type { ApiResponse, PageParams, PageResult } from '#nuva/api'
```

`#nuva/api` 适合前端 API client 和页面侧类型；`shared/api` 只放 Nuxt server 和前端都需要共享的 schema/type。

## 业务 API 组织方式

推荐把业务请求放在：

```txt
app/composables/apis/*.ts
```

同一业务域尽量收敛到一个文件：

- `useXxxApi()` 放接口方法。
- `useXxxTable()` 放分页、筛选等状态组合。
- 页面只负责调用，不直接处理 baseURL、token、响应解包和错误协议。

模板提供了轻量起步 helper，适合常见中后台 CRUD：

```ts
export function useUserApi() {
  return createResourceApi<User, UserCreateInput, UserUpdateInput, UserQuery>({
    path: '/users',
  })
}

export function useUserTable() {
  const query = reactive<UserQuery>({ keyword: undefined, status: undefined })

  return {
    query,
    ...usePageApi<User, UserQuery>(useUserApi().list, { query }),
  }
}
```

`createResourceApi()` 和 `usePageApi()` 是模板可覆盖包装；core 中对应稳定入口为 `createNuvaResourceApi()` 和 `useNuvaPageApi()`。复杂业务动作仍建议在 `useXxxApi()` 中直接手写 alova Method。

## OpenAPI 生成

OpenAPI 生成使用可选模块 `@oevery/nuva/openapi`，基于 `@alova/wormhole`，默认输出到 `app/api/generated`。生成代码是前端 API client，不放入 `shared/api`。

启用模块后也可以从 `#nuva/openapi` 导入生成入口：

```ts
import { useGeneratedApis } from '#nuva/openapi'
```

## 单接口覆盖

单个接口可以通过 alova `meta` 覆盖全局行为：

```ts
http.Get('/profile', {
  meta: {
    ignoreToken: true,
    responseType: 'json',
    envelopeUnwrap: true,
    successCodes: '0,200',
  },
})
```

完整 meta 字段见 [HTTP API 参考](/reference/http)。

## 错误处理边界

普通接口错误在请求 hooks 中处理；登录态失效和 401/403 的行为由 Auth 配置和 `useDefaultHttpRequestHooks()` 统一处理。

模板可以在 `app/utils/http/hooks.ts` 中追加请求头、traceId、租户和错误提示 UI。
