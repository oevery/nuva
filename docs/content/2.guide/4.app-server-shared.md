---
title: App / Server / Shared
description: 用一套简单规则判断业务代码应该放在 app、server 还是 shared。
---

Nuva 推荐沿用 Nuxt 的全栈目录模型，并把业务协议放进 `shared`。这能减少“页面一份类型、接口一份类型、服务端又一份校验”的漂移。

## `app` 负责用户可见部分

放在 `app` 的内容通常与浏览器交互相关：

- 页面和路由。
- 表单、列表、详情、布局。
- 前端 API composable。
- 请求 hooks 和 UI 错误提示。

业务页面不要直接拼接响应协议，应该调用 `app/composables/apis` 中的封装。

## `server` 负责服务端能力

放在 `server` 的内容运行在 Nuxt server：

- `server/api` 接口。
- `server/utils` 服务端工具。
- 认证实例、数据库连接、server adapter。

服务端接口必须自己做权限和数据校验，不能只依赖前端页面保护。

## `shared` 负责共同协议

放在 `shared` 的内容必须能被前端和服务端同时引用：

- 业务实体类型。
- 请求和响应类型。
- valibot schema。
- 分页、枚举、通用协议。

不要把服务端私有结构或单页面临时展示类型放进 `shared`。

## 判断规则

| 问题 | 放置位置 |
| --- | --- |
| 用户能看到或操作吗？ | `app` |
| 只在服务端运行吗？ | `server` |
| 前端和服务端都需要吗？ | `shared` |
| 是 Nuva 请求/认证/权限配置吗？ | `nuva.config.ts` |
| 是 Nuxt 模块或构建行为吗？ | `nuxt.config.ts` |
